# 09 · Internacionalización (i18n)

> **Alcance:** locales soportados, librería usada, estructura de claves, formato de fechas/números y reglas para añadir traducciones.
> Para cómo se persiste la preferencia, ver `07-state-and-data.md`. Para tipografías, ver `08-styling-and-tokens.md`.

## Locales soportados

| Locale | Default | Bundle |
|---|---|---|
| `es-MX` | ✓ | `src/messages/es-MX.json` |
| `en-US` |   | `src/messages/en-US.json` (fallback parcial a `es-MX`) |

Ambos heredan del prototipo (`app/i18n.jsx` declara más de 200 claves en `es-MX` y un bundle `en-US` opcional).

## Librería: `next-intl`

Elegida por:

- Soporte nativo de Server Components.
- Mensajes en JSON sin compilación adicional.
- Formato ICU MessageFormat (plurales, género, fechas, números) — necesario para `{N} clientas`, `1 cita / N citas`.
- API de cliente (`useTranslations`) idéntica a la de servidor.

Setup (resumen):

```ts
// src/middleware.ts (orden con auth en 06-routing-and-rbac.md)
import createIntlMiddleware from "next-intl/middleware";
import { locales, defaultLocale } from "@/config/i18n";

export default createIntlMiddleware({
  locales,
  defaultLocale,
  localeDetection: true,
  localePrefix: "as-needed",     // /es-MX/... opcional, default sin prefijo
});
```

```ts
// src/config/i18n.ts
export const locales = ["es-MX", "en-US"] as const;
export const defaultLocale = "es-MX" as const;
export type Locale = (typeof locales)[number];
```

## Reglas para añadir traducciones

1. **Una clave = un texto**. No reutilizar la misma clave con interpolaciones diferentes.
2. **Inglés en el código, español en `messages/es-MX.json`**. Si el botón es "Registrar venta", el código pone `<button>{t("profile.actions.register_sale")}</button>`, **nunca** `<button>Registrar venta</button>`.
3. **Plurales con ICU**: `{count, plural, one {...} other {...}}`. Cubre `=0` cuando hay un caso vacío especial.
4. **Variables**: `{name}`, `{count}`, `{date, date, short}`. Nombres en `snake_case`.
5. **No concatenar strings** desde el código (`"Hola " + name`); siempre interpolación en la clave.
6. **No traducir nombres de marca** (Lancôme, YSL, Liverpool, Palacio).

## Uso desde Server Components

```tsx
// src/app/(app)/ba/clients/page.tsx
import { getTranslations } from "next-intl/server";
import { ClientList } from "@/features/clients/components/client-list";

export default async function ClientsPage() {
  const t = await getTranslations("clients");
  return (
    <main>
      <h1>{t("title")}</h1>
      <ClientList />
    </main>
  );
}
```

## Uso desde Client Components

```tsx
"use client";
import { useTranslations } from "next-intl";

export function SegmentChips() {
  const t = useTranslations("clients.segment");
  const options = ["all", "vip", "recurrent", "new", "at_risk"] as const;
  return options.map(k => <Chip key={k}>{t(k)}</Chip>);
}
```

## Tipado de claves

`next-intl` admite type-safety a partir del JSON:

```ts
// src/global.d.ts
import type messages from "./messages/es-MX.json";
declare global {
  interface IntlMessages extends typeof messages {}
}
```

Esto hace que `t("profile.tab.timeline")` autocomplete y falle en compilación si la clave no existe.

## Formato de fechas y números

`next-intl` usa la API nativa `Intl`. Los helpers actuales `MXN`, `fmtDate`, `fmtTime`, `fmtRel` (de `app/data.jsx`) se reemplazan por wrappers que delegan al locale activo:

```ts
// src/lib/format/format-currency.ts
import { useFormatter } from "next-intl";
export function useCurrency() {
  const f = useFormatter();
  return (amount: number) => f.number(amount, { style: "currency", currency: "MXN" });
}
```

Equivalente en servidor:

```ts
import { getFormatter } from "next-intl/server";
const f = await getFormatter();
f.number(123456, { style: "currency", currency: "MXN" });
```

| Helper actual | Reemplazo |
|---|---|
| `MXN(n)`, `MXNc(n)` | `useCurrency()` / `getFormatter().number(n, { style: "currency", currency: "MXN" })` |
| `fmtDate(iso)` | `formatter.dateTime(date, { dateStyle: "long" })` |
| `fmtTime(iso)` | `formatter.dateTime(date, { timeStyle: "short" })` |
| `fmtRel(iso)` | `formatter.relativeTime(date, now)` |

## Idioma del cliente vs idioma de la app

Existen **dos** preferencias distintas:

| Preferencia | Persistencia | Afecta |
|---|---|---|
| Locale de la app (UI del BA/Manager/etc.) | Cookie `locale` (next-intl) | Strings, formatos en la app |
| `client.preferredLang` (del cliente) | Campo en el agregado `Client` | Plantillas de mensajes salientes (`features/communications/`) |

El compositor de WhatsApp / Email usa `client.preferredLang` para escoger la plantilla, no el locale de la app. Eso evita que el BA en inglés envíe templates en inglés a una clienta hispanohablante.

## Selector de idioma

- **No** existe en TopBar (decisión heredada del prototipo: el idioma es preferencia personal del BA).
- Vive en `/settings/profile` cuando se implemente.
- Mientras tanto, `next-intl` lo determina por header `Accept-Language`.

## Cargar bundles

`next-intl` carga el bundle correspondiente por locale automáticamente:

```ts
// src/i18n.ts
import { getRequestConfig } from "next-intl/server";

export default getRequestConfig(async ({ locale }) => ({
  messages: (await import(`./messages/${locale}.json`)).default,
}));
```

Code-splitting automático: solo se envía el bundle del locale activo al cliente.

## Migración del módulo actual

| Concepto en `app/i18n.jsx` | Reemplazo |
|---|---|
| `window.LxI18n.t(key, params)` | `t(key, params)` desde `next-intl` |
| `window.LxI18n.setLang(lang)` | Cambio de cookie + `router.refresh()` |
| Evento `lx-i18n` | `<NextIntlClientProvider>` propaga automáticamente |
| `localStorage` `lx-clienteling:v1:lang` | Cookie `NEXT_LOCALE` (manejada por next-intl) |
| Fallback en código | Fallback declarativo en `messages/en-US.json` (claves vacías heredan de `es-MX.json`) |
